# Token proposal: missing beta colour tokens (#5221)

Import-ready Tokens Studio JSON closing the gaps the Button migration
([#5208](https://github.com/equinor/design-system/issues/5208)) surfaced in
the beta colour export. Part of
[#4740](https://github.com/equinor/design-system/issues/4740); token content
only — the `:root`-scoping export problem is tracked separately in
[#5226](https://github.com/equinor/design-system/issues/5226).

The three JSON files are **complete replacement sets** for their
counterparts in `src/tokens/raw/`. They are generated by
`build-proposal.mjs` from the current pull, so the diff against
`src/tokens/raw/` is exactly the proposal:

```bash
node build-proposal.mjs                             # regenerate (idempotent)
git diff --no-index ../../src/tokens/raw/semantic.json semantic.json
```

## Changes

### 1. Alpha step twins — `scheme/light.json`, `scheme/dark.json`

New groups `accent-alpha`, `neutral-alpha`, `danger-alpha` with steps `3`
and `7`, as **explicit hex8 values** (the platform only evaluates colour
formulas in the CSS export, so `modify` formulas cannot be used). These are
the alpha-channel disabled colours from
[#4741](https://github.com/equinor/design-system/issues/4741): step 3 is
the disabled fill, step 7 the disabled text/border/icon — the same steps
the legacy bundle and the scheme sets already used for disabled states
(reuse of existing values per
[#4742](https://github.com/equinor/design-system/issues/4742), step-based
naming per [#4744](https://github.com/equinor/design-system/issues/4744)).

Each twin composites to its opaque step **exactly** over white (light
scheme) / black (dark scheme) — the standard alpha-scale construction,
which keeps alpha minimal. Over the actual canvas (`neutral.1`) the
composite deviates by the few 8-bit steps listed below; adapting to the
surface underneath is the point of the alpha approach.

Colour math matches `scripts/generate-ts-tokens.mjs` (oklch → sRGB with
CSS Color 4 gamut mapping), and compositing happens on gamma-encoded 8-bit
channels, which is how browsers composite `rgba` over opaque backgrounds.
The high dark `danger-alpha.7` alpha is inherent: saturated bright colours
over black need high opacity (Radix dark red alphas behave the same).

### 2. Semantic fill group — `semantic.json`

`background.fill.{accent,neutral,danger}.{default,hover,pressed,disabled}`
→ steps `9`/`10`/`11` + `{tone}-alpha.3`.

There is no semantic token for solid action fills today
(`background.surface.accent.*` = steps 14/11/15, unusable as a fill), so
the migrated Button consumes steps 9/10/11 directly. This group gives
those choices a semantic home; the steps match the legacy fill values
exactly.

### 3. Surface tone equivalents — `semantic.json`

`background.surface.{neutral,danger}` mirror the existing accent-only
group step for step (`default` 14/11/15, `selected` 5/6/7, `subtle`
2/4/6). Components with tone support (Button, Chip) need the same
surfaces in neutral and danger.

### 4. Tone text and border — `semantic.json`

- `text.{accent,neutral,danger}` → step `12`; `border.{accent,neutral,danger}`
  → step `8` (what secondary/ghost Button consumes today).
- `text.accent` existed but pointed at `{info.8}` (link blue) — **repointed
  to `{accent.12}`**. Link blue remains available as `text-link` in the
  scheme sets.
- `text.neutral` duplicates `text.primary` by value; it exists so the three
  tones can be switched uniformly (`--eds-text-accent` ↔ `-neutral` ↔
  `-danger`).

### 5. Name-collision fixes — one owner per CSS name

`border-focus`, `border-disabled` and `text-disabled` were declared by
**both** the semantic set and the scheme sets, flattening to the same
`--eds-*` names. `--eds-border-focus` was a self-reference (invalid
outside a scheme subtree), and `border-disabled`/`text-disabled` had two
conflicting values (semantic said `neutral.2`/`neutral.5`, scheme said
`neutral.7`). The semantic layer becomes the single owner:

- `border.focus` → `{info.7}` — resolves to the exact values the removed
  concept token had (`light.blue.7` / `dark.blue.7`).
- `border.disabled`, `text.disabled`, `icon.disabled` → `{neutral-alpha.7}`
  — the step-7 value the scheme sets and the legacy bundle used, now as an
  alpha twin.
- The scheme sets drop `border-focus`, `border-disabled`, `text-disabled`.

## Open questions for design

1. **Alpha percentages** need sign-off to become canonical (they are
   derived, not designed).
2. **Disabled strength**: the semantic set previously said `neutral.5`
   (text/icon) and `neutral.2` (border); the scheme sets and the legacy
   bundle said step 7. This proposal standardises on step 7 — confirm.
3. **`text.accent` repoint** from link blue to `{accent.12}` — confirm the
   link-blue use cases are covered by `text-link`.
4. Only `accent`/`neutral`/`danger` get alpha twins and tone groups here;
   `info`/`success`/`warning` can follow the same recipe when a component
   needs them.

## Importing (after design review)

1. In the Tokens Studio UI, create a **work branch** — the platform has no
   undo.
2. Import each JSON file as a **replacement** for its set (`semantic`,
   `scheme/light`, `scheme/dark`). The collision fixes are removals — a
   merge that only adds tokens will miss them.
3. Design review on the branch, merge, cut a beta release.
4. After `studio tokens pull`, verify `src/tokens/css/` resolves:
   `--eds-border-focus` no longer self-references, the `--eds-*-alpha-*`
   variables exist under both scheme scopes, and
   `--eds-background-fill-accent-default` resolves to `accent.9`.
